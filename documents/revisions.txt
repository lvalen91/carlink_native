[31] :: FIX: Progressive video quality degradation.
   - Root cause: PacketRingByteBuffer.readPacket() used zero-copy ByteBuffer.wrap() sharing memory with ring buffer
   - Race condition allowed USB writes to overwrite data before MediaCodec consumed it, corrupting H.264 reference frames
   - Fix: Safe copy via System.arraycopy() before returning ByteBuffer
   - Periodic keyframes didn't help because they were also corrupted; only reset() worked (cleared buffer first)

[56] :: MAJOR RESTRUCTURE: Projection Video Model. "Buffer for stability, not completeness." CarPlay/Android Auto is live interactive UI, not cinema playback.
       BUFFER: Reduced from 2-4MB to 192KB jitter buffer. Packet limit 120→12 (~200ms).
       STALENESS: Frames >150ms old discarded at entry. Skip-to-newest-IDR on overflow.
       PTS: Monotonic synthetic only. Source PTS queue removed.
       QC: Quality Control system removed entirely (was counterproductive).

       USB CAPTURE ANALYSIS: Confirmed adapter sends SPS+PPS+IDR at start, periodic IDRs (~1.4s) on request.
[57] :: CRITICAL FIX: Decoder poisoning from staleness time base bug.
   - Root cause: staleness check compared epoch time (~1.7T ms) to stream-relative PTS (~32K ms)
   - Incompatible time bases made all frames appear "stale", IDR frames discarded, preventing decoder recovery
   - Fix 1: Disabled staleness check entirely (harmful, not just incomplete)
   - Fix 2: Reduced STALL_TIMEOUT_MS 500ms to 200ms for faster recovery
   - Fix 3: Added corruption detection via IDR tracking (IDRs decoded vs frames output, triggers reset if poisoned)
   - USB capture verified adapter sends correct data (88 IDRs/169s, all bundled with SPS+PPS)
   - Confirmed PTS is stream-relative, not epoch. Average adapter frame delay: 7.85ms
